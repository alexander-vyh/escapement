escapement

OpenSpec, opinionated.

A snapshot of one working Claude Code setup. Built on top of OpenSpec, wired to Beads, enforced by Claude Code hooks. Strong opinions. Take what's useful, ignore the rest.

github.com/alexander-vyh/escapement

02

What this actually is

A workflow built on OpenSpec, not a replacement. Four overlays:

01

Adversarial discovery

Riskiest-assumption, pre-mortem, red/blue team, walking-skeleton — injected as internal prompts; not visible in the output docs.

02

Bridge to beads

OpenSpec's tasks.md becomes a real task graph via bd create --spec-id …. Agents execute against acceptance criteria linked back to specs.

03

Molecule formula

mol-feature orchestrates brainstorm → discovery → skeleton → build → retro as a single bd mol pour command with two human gates.

04

Discipline hooks

Tool-call-level enforcement: TDD-first, named-agent teams, outcome verification, spec-ID linkage, never-suppress.

The artifacts on disk — proposal.md, design.md, specs/*.md, tasks.md — are standard OpenSpec. A teammate using bare OpenSpec can read them with zero extra context.

03

The core loop

you say "build X"
A user prompt with feature-intent keywords
↓
mol-feature (beads)
Root epic + 9 step tasks (2 are gates) created via bd mol pour
↓ step: discovery
/discovery skill
Adversarial overlay on OpenSpec CLI → writes openspec/changes/{name}/{proposal,design,specs/*,tasks}.md
↓ step: work-breakdown
/work-breakdown skill
Reads openspec/, writes bd spec issues (--acceptance) and bd task issues (--spec-id linked)
↓ step: execute-skeleton → review → execute-full
parallel named agents
Implement tasks, reading --acceptance via --spec-id. Each agent gates on a Test Oracle Brief before writing code.

05

What a mol-feature pour produces

Five concrete moments, in order.

1
You pour the molecule
bd mol pour mol-feature \
  --var name="add-status-cmd" \
  --var problem="agents need a /status slash command"
2
Beads creates an epic + 9 step tasks, with dependencies wired up
repo-abc        [epic]    add-status-cmd
repo-abc.1      [step]    brainstorm           ready
repo-abc.2      [step]    discovery            blocked
repo-abc.3      [gate]    review-discovery     blocked
repo-abc.4      [step]    work-breakdown       blocked
repo-abc.5      [step]    execute-skeleton     blocked
repo-abc.6      [gate]    review-skeleton      blocked
repo-abc.7      [step]    execute-full         blocked
repo-abc.8      [step]    ceremony-retro       blocked
repo-abc.9      [step]    outcome-check        blocked
3
Next prompt: mol_status_check fires on SessionStart
Active molecule: add-status-cmd  (Design phase)
Current step: brainstorm  [ready]
Next: run /brainstorm or /discovery to start.
4
Artifacts accumulate as each step completes
after discovery        openspec/changes/add-status-cmd/
                         ├─ proposal.md
                         ├─ design.md
                         ├─ specs/{capability}.md
                         └─ tasks.md

after work-breakdown   repo-abc.spec.*   bd spec issues (--acceptance)
                       repo-abc.task.*   bd task issues (--spec-id linked)

after execute-full     production code + passing tests
                       all task issues closed
5
Two gates pause for your approval

review-discovery — design.md and specs are ready: approve, or revise?
review-skeleton — walking skeleton validated the riskiest assumption: proceed to full build, or back to design?

07

The skill stack

Eight skills. Each wraps a step of the workflow.

  • build — front-door router; classifies the work, pours the right molecule
  • brainstorming — "should we build this at all?" pre-filter with rotating creative lenses
  • discovery — adversarial wrapper on OpenSpec CLI; injects pre-mortem, red/blue team, walking-skeleton
  • work-breakdown — reads openspec/changes/{name}/, creates bd spec + task issues with spec-id links
  • beads-execution — reads bd ready, dispatches named-team subagents per task
  • behavioral-test-oracle-review — required Test Oracle Brief before any behavioral code change
  • dispatching-parallel-agents — mandatory TeamCreate + named agents; no anonymous dispatch
  • subagent-driven-development — two-stage review per task: spec compliance → code quality
08

The rules — opinions, made explicit

  • planning-discipline.md — pour mol-feature for any non-trivial work
  • tdd-enforcement.md — failing test FIRST in any test-capable repo
  • agent-teams-default.md — TeamCreate + named agents for anything multi-step
  • outcome-ownership.md — done = verified end-to-end, not "compiles"
  • molecule-awareness.md — surface active molecules on every session start
  • beads-worktree-integration.md — bd worktree create, not git worktree add
  • never-suppress.md — no #noqa, no --no-verify, no test downgrades
  • serena-first.md — symbol tools over full-file Read when Serena is onboarded

Read each before installing. Edit to match your philosophy. These are not universally applicable — TDD and never-suppress in particular will feel restrictive in repos with legacy suppression patterns.

09

The hooks — what the rules can't enforce alone

OpenSpec workflow

  • openspec_init_guard — block openspec commands when openspec/ missing
  • design_doc_location_guard — warn on docs/plans/ design docs
  • discovery-gate — block bd create of features without design doc
  • discovery-nudge — nudge /discovery on feature-intent prompts
  • discovery-close-gate — surface proof-of-delivery on bd close
  • spec_id_enforcement — require --spec-id under mol-feature
  • mol_status_check — surface active molecules on session start

TDD + test oracle discipline

  • tdd-gate — require test edits before implementation
  • test_oracle_brief_gate — require Test Oracle Brief for code changes
  • test_reminder — PostToolUse nudge to run tests
  • implementation_echo_test_gate — reject tests that echo implementation
  • oracle_downgrade_warning_gate — warn on test oracle weakening
  • outcome_assertion_gate — on PR create, gate on assertion quality

Outcome verification + shirking

  • validate_no_shirking — block "pre-existing failure" evasion
  • review_gate — soft gate on bd close without review agent
  • review_nudge — nudge /review on review-intent prompts

Agent discipline

  • enforce_named_agents — block anonymous or teamless multi-agent dispatch
  • context_burn_detector — nudge agent dispatch after excessive inline research
  • session_cleanup — clean /tmp state from the above

Serena navigation (silent unless onboarded)

  • serena_preference_gate — block full-file Read when Serena available
  • serena_preference_injection — UserPromptSubmit guidance toward symbol tools
  • serena_onboarding_check — nudge onboarding when missing
10

What to borrow

Not interested in the whole stack? Here's the ordered shopping list.

If you take ONE thing
The adversarial discovery overlays in claude/skills/discovery/SKILL.md. Plain OpenSpec discovery, plus internal prompts for riskiest-assumption, pre-mortem, red/blue team, walking-skeleton. Output looks like normal OpenSpec — only the thinking that produced it changed.
If you take TWO
+ the spec-ID linkage in claude/skills/work-breakdown/SKILL.md and claude/hooks/spec_id_enforcement.py. Connects OpenSpec specs to beads tasks so agents working a task can read its acceptance criteria directly. The hook makes it impossible to forget.
If you take THREE
+ never-suppress.md and outcome-ownership.md. Two short discipline rules. No #noqa, no --no-verify, no test downgrades; "done" means the user's outcome is happening, not "compiles." Drop into ~/.claude/rules/ and they're live.
If you take EVERYTHING
Clone the repo, run ./INSTALL.sh, merge settings.template.json. Symlink-based install — git pull updates your live config. Existing files backed up to *.backup-{timestamp} first.
11

Warnings

This is opinionated. The rules will feel restrictive if your project lacks test infrastructure or relies on legacy suppression patterns. Read claude/rules/ before installing. Edit to match your philosophy.
Hooks are load-bearing. Skills produce guidance. Hooks produce enforcement. Skills without hooks → Claude drifts. Hooks without skills → blocks without a path forward. Install both, or neither.
Path assumptions. Bootstrap script is gated on ~/GitHub/*. Some rules reference ~/GitHub/ paths. Edit if your repos live elsewhere.
Personal config not bundled. ~/.claude/settings.json (auth, permissions) and ~/.claude/projects/*/memory/ are excluded. The installer manages only the workflow files.

Try it

git clone https://github.com/alexander-vyh/escapement ~/GitHub/escapement
cd ~/GitHub/escapement
./INSTALL.sh
# then manually merge claude/settings.template.json into ~/.claude/settings.json

Validation prompt for the first session

Open Claude Code in a fresh git repo under ~/GitHub/ and say:

let's build a small feature to validate the setup — add a /status slash command

Claude should pour mol-feature and walk you through brainstorm → discovery → review → breakdown → skeleton → build → retro. If that flow happens end-to-end, you're wired correctly.

Uninstall

./INSTALL.sh --uninstall   # removes symlinks; backups left alone
